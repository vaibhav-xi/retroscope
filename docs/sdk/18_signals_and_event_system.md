# 18 - Signals & Event System

Version: 1.0

---

# Introduction

As Retroscope grows, modules must communicate with one another without becoming tightly coupled.

For example

```
Audio Module

↓

Beat Detected

↓

Particle Module

↓

Emit Burst
```

or

```
Preset Loaded

↓

Every Module Updates
```

Rather than allowing modules to call each other directly, Retroscope uses a **Signal (Event) System**.

Signals allow one part of the engine to announce that something happened.

Other parts of the engine may choose to react.

---

# Philosophy

Signals answer

> **"Something happened."**

They do **not** answer

> "What should everyone do?"

This distinction is important.

A signal is a notification.

The receivers decide how to respond.

---

# Why Signals?

Imagine there are twenty visualization modules.

Without signals

```
Module A

↓

Module B

↓

Module C

↓

Module D

↓

...
```

Every module eventually depends on every other module.

The architecture becomes impossible to maintain.

Signals instead create

```
Emitter

↓

Signal

↓

Subscribers
```

No direct references exist.

---

# Event Flow

Conceptually

```
Something Happens

↓

Signal Emitted

↓

Dispatcher

↓

Subscribers

↓

Reaction
```

The emitter never knows who received the event.

---

# Ownership

The Signal System belongs to the engine.

```
Application

↓

Signal Dispatcher

↓

Modules
```

Modules never own the dispatcher.

---

# Lifetime

```
Application Starts

↓

Dispatcher Created

↓

Events Flow

↓

Application Exits
```

The dispatcher exists for the application's lifetime.

# Communication Rules

Signals should communicate

```
Events
```

not

```
State
```

State belongs in Context.

Example

Good

```
ThemeChanged
```

Bad

```
CurrentTheme = Blue
```

The module can read the current theme from Context.

---

# Relationship With Context

Context stores

```
Current State
```

Signals announce

```
State Changed
```

These systems complement each other.

---

# Threading

Signals should generally be dispatched on the main thread.

Background services

(audio, networking)

may queue events for later dispatch.

---

# Performance

Signals should be lightweight.

Events such as

```
BeatDetected
```

occur frequently.

Dispatch should avoid unnecessary allocations.

---

# Best Practices

✔ Emit meaningful events.

✔ Subscribe only when necessary.

✔ Keep callbacks short.

✔ Read detailed state from Context.

✔ Unsubscribe when appropriate.

---

# Anti-Patterns

Avoid

```
Module A

↓

Calls Module B
```

Avoid

```
Global Variables
```

Avoid

```
Polling Every Frame

↓

Did Theme Change?

Did Preset Change?

Did Audio Start?
```

Signals exist specifically to eliminate unnecessary polling.

# Mental Model

Imagine a fire alarm.

The alarm doesn't know

- who hears it
- who evacuates
- who calls emergency services

It simply announces

> **"Fire!"**

Everyone reacts according to their own responsibilities.

Signals work exactly the same way.

---

# Summary

The Signal System provides Retroscope with a lightweight, decoupled communication mechanism.

Instead of allowing modules and services to depend directly on one another, events are broadcast through a central dispatcher and handled only by interested subscribers.

Context represents **what the engine currently is**, while Signals represent **what has just happened**.

Together they form the backbone of communication inside Retroscope and enable the engine to scale without creating tightly coupled subsystems.
